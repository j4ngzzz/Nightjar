What is Vericoding?

Vericoding is LLM-generation of formally verified code from formal specifications. The output is not code that looks right — it is code accompanied by a mathematical proof that it satisfies an explicit contract.

Term coined in: arXiv:2509.22908 · Bursuc, Ehrenborg, Lin et al. (Beneficial AI Foundation / MIT, 2025)

The definition

“Vericoding: LLM-generation of formally verified code from formal specifications.”
— Bursuc, Ehrenborg, Lin et al., “A benchmark for vericoding: formally verified program synthesis”, arXiv:2509.22908, Beneficial AI Foundation / MIT (2025)

The BAIF benchmark evaluated AI systems across 12,504 formal specifications in three verification languages — Dafny, Verus/Rust, and Lean. LLMs achieved 82% success in Dafny, 44% in Verus/Rust, and 27% in Lean. Pure Dafny verification success improved from 68% to 96% in a single year, demonstrating that the practice is not just theoretically sound — it is industrially tractable today.

Vericoding is not a niche academic exercise. It is the logical endpoint of AI-assisted development: if an LLM is writing your code, you can also require that LLM to supply a proof of correctness rather than trusting your test suite to catch every edge case.

Vibe coding vs. Vericoding

The difference is not speed — it is the nature of the guarantee you get at the end.

DimensionVibe CodingVericoding
SpecificationNatural language promptFormal machine-checkable spec
Correctness guaranteeNone — code might workMathematical proof
Verification methodHuman review, ad-hoc testsAutomated verifier (Dafny, Lean, Verus)
Output artifactCode + test suiteCode + proof + audit trail
Failure recoveryPrompt again, guessCEGIS — counterexample-guided repair
Manual editingRequired — AI output is a draftProhibited — spec is the source of truth
Risk profileHigh — unknown unknownsBounded — only unspecified behaviors can fail

How Nightjar implements vericoding

Nightjar is a contract-anchored regenerative development pipeline. Developers write specs. AI generates code. Nightjar proves the code satisfies the specs. Code is never manually edited.

Stage 0Preflight

Parse .card.md specs. Run 19 proven rewrite rules to normalise invariants before generation.

Stage 1Dependency Audit

Seal the dependency manifest. Run pip-audit and uv to detect vulnerabilities in the supply chain.

Stage 2Schema Validation

Validate all data structures against Pydantic v2 models derived from the spec.

Stage 2.5Negation Proof

Prove that the negation of each invariant is unsatisfiable — an early counterexample check.

Stage 3Property-Based Testing

Run Hypothesis PBT strategies generated from the spec. Failures become CEGIS counterexamples.

Stage 4Dafny Formal Proof

Translate the spec into Dafny and call the verifier. A CEGIS retry loop repairs failing proofs automatically.

Frequently asked questions

What is vericoding?

Vericoding is the practice of using LLMs to generate formally verified code from formal specifications. The generated code is not merely tested — it is accompanied by a mathematical proof that it satisfies the spec. The term was coined in the 2025 BAIF/MIT benchmark paper (arXiv:2509.22908), which evaluated AI systems across 12,504 formal specifications in Dafny, Verus/Rust, and Lean.

How is vericoding different from vibe coding?

Vibe coding uses natural language prompts and intuition to generate code, accepting informal or no specification and no correctness guarantee. Vericoding starts with a formal machine-checkable spec, then generates code that an automated verifier proves satisfies that spec. The output is not code that looks right — it is code that is provably correct against an explicit contract.

What tools support vericoding today?

The three verification languages benchmarked in arXiv:2509.22908 are Dafny (82% LLM success rate), Verus/Rust (44%), and Lean (27%). Nightjar adds a full pipeline around Dafny: it reads .card.md specs, generates code via an Analyst→Formalizer→Coder chain, and runs a 6-stage pipeline ending in a Dafny formal proof.

Does vericoding slow down development?

For AI-generated code specifically, vericoding can be faster than manual debugging. The BAIF benchmark showed pure Dafny verification success improved from 68% to 96% in one year, meaning the overhead of writing a spec is increasingly offset by the LLM generating correct code on the first attempt. Nightjar's CEGIS retry loop automates recovery from verification failures without human intervention.

What is the BAIF benchmark for vericoding?

The BAIF (Beneficial AI Foundation) benchmark, arXiv:2509.22908, is a dataset of 12,504 formal specifications: 3,029 in Dafny, 2,334 in Verus/Rust, and 7,141 in Lean. It was produced by Sergiu Bursuc, Theodore Ehrenborg, Shaowei Lin, and 10 co-authors including Max Tegmark (MIT). It is the first large-scale benchmark specifically for formally verified program synthesis by LLMs.

How does Nightjar implement vericoding?

Nightjar implements a 6-stage vericoding pipeline: Stage 0 (preflight), Stage 1 (dependency audit), Stage 2 (schema validation via Pydantic), Stage 2.5 (negation proof), Stage 3 (property-based testing via Hypothesis), and Stage 4 (Dafny formal proof). Specs are written in .card.md files. Code is regenerated from scratch on every build and never manually edited. A CEGIS retry loop automatically repairs failing specs.
